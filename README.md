# 뷁어번역기 Pro

> 인코딩 깨짐(이른바 **뷁어**, mojibake)을 일본어·한국어·중국어·UTF-8 등으로
> **양방향 변환**하는 서버리스 정적 웹앱. 깨진 글을 붙여넣으면 원본을 추론하는
> **스마트 복원**과 레트로 Windows XP 창 UI를 갖췄습니다.

🔗 **데모:** https://jtech-co.github.io/BWEK-Pro/

---

## 뷁어란?

한 인코딩으로 저장한 바이트를 **다른 인코딩으로 잘못 읽으면** 글자가 깨집니다.
대표적으로 일본어(Shift-JIS) 텍스트의 바이트를 한국어(CP949)로 읽으면 정체불명의
한글 덩어리가 나오는데, 이를 **뷁어**라고 부릅니다.

```
あいうえお  ──(Shift-JIS 바이트를 CP949로 읽음)──▶  궇궋궎궑궓
궇궋궎궑궓  ──(CP949 바이트를 Shift-JIS로 읽음)──▶  あいうえお
```

## 기능

- **다방면 상호 변환** — 일본어(Shift-JIS/EUC-JP)·한국어(CP949/EUC-KR)·중국어(GBK/Big5)·UTF-8·UTF-16·Latin-1·Windows-1252 사이의 임의 조합 변환
- **프리셋** — 일↔뷁, 한↔뷁, UTF-8↔Latin-1(웹 고전 깨짐), UTF-8↔CP949(파일 깨짐), 중국어↔뷁 등 자주 쓰는 시나리오 + **사용자 지정**(인코딩 직접 선택)
- **스마트 복원** — 깨진 글을 붙여넣으면 여러 오독 조합을 시도해 가장 자연스러운 후보를 신뢰도순으로 제시
- **레트로 XP UI** — 첨부 스크린샷의 창 스타일을 충실히 복원하되 접근성(라벨/포커스/대비/반응형)을 개선
- **서버리스** — 모든 변환이 브라우저에서 일어나며 **입력 내용이 서버로 전송되지 않음**
- **공유 링크** — 현재 입력·변환 설정을 URL 해시에 담아 그대로 공유
- **부가 기능** — 실시간 변환, 입력↔결과 스왑, 복사, 글자수/바이트 표시

## 변환 원리

모든 깨짐/복원은 단 한 줄로 표현됩니다.

```
reinterpret(text, from, to) = decode(to, encode(from, text))
```

- **디코딩(바이트→문자)** 은 브라우저 내장 `TextDecoder` 를 그대로 사용합니다.
  브라우저의 `euc-kr` 디코더는 WHATWG 표준에 따라 **CP949/UHC 슈퍼셋**이라 한글 전 영역을 처리합니다.
- **인코딩(문자→바이트)** 은 표준 API가 UTF-8만 지원하므로, 같은 인코딩의
  `TextDecoder` 를 런타임에 **역으로 훑어 "문자→바이트" 표**를 만들어 캐시합니다.

덕분에 **인코딩 데이터 테이블을 전혀 번들하지 않아** 런타임 의존성이 0이고,
빌드 결과물이 ~10 KB(gzip)에 불과합니다.

## 로컬 실행

```bash
npm install
npm run dev       # 개발 서버 (http://localhost:5173)
npm run build     # 타입체크 + 프로덕션 빌드 (dist/)
npm run preview   # 빌드 결과 미리보기
```

요구사항: Node.js 20.19+ 또는 22.12+.

## 배포

`main` 브랜치에 push 하면 [GitHub Actions](.github/workflows/deploy.yml)가 빌드 후
**GitHub Pages** 로 자동 배포합니다. 프로젝트 사이트 경로(`/BWEK-Pro/`)는
CI 빌드에서 자동 적용됩니다.

## 한계

특정 글자는 양쪽 인코딩에서 모두 유효한 바이트가 아니라 **왕복이 손실**될 수 있습니다.
예컨대 일부 한자는 Shift-JIS 하위 바이트가 CP949에 존재하지 않아 복원되지 않습니다 —
이는 버그가 아니라 **실제 뷁어에서도 동일하게 나타나는 현상**입니다. 가나·한글처럼
양쪽에서 유효한 글자는 완벽히 왕복됩니다.
